# Wakaba — авторинг контента: соглашения

> Правила именования и структуры для `content/`. Ими руководствуются автор и ИИ-ассистенты.
> Машинно-проверяемая часть закреплена в утилите (`npm run content -- check`); остальное — здесь.

## 1. id — главный принцип

- **Вечный.** id не переименовывают: в БД это новая сущность, а старые ссылки и избранное пользователей ломаются. Выбирай имя сразу правильно. Нужно «переименовать» → заводи новый id и мигрируй осознанно.
- **kebab-case** везде (`ask-speak-english`, `wa-note`). Токены — это ромадзи слова, обычно одно слово без дефисов и **без подчёркиваний** (`hanasemasu`, `konnichiwa`).
- **Глобально уникальный по всем типам.** Один id = ровно одна сущность (токен ИЛИ заметка ИЛИ фраза…). Валидатор падает на коллизии.

## 2. Именование по типам

| Сущность | Правило id | Примеры |
|---|---|---|
| **token** | ромадзи слова; омоним → суффикс смысла | `eigo`, `hanasemasu`, `wa`, `ka`, `ka-or`, `no-possessive` |
| **token-выражение из нескольких слов** | kebab-case по ромадзи; учится как одно целое | `arigatou-gozaimasu`, `yoroshiku-onegaishimasu` |
| **grammar-note о токене** | `<token-id>-note` | `wa-note`, `ka-note`, `dake-note`, `konnichiwa-note`, `sukoshi-note` |
| **grammar-note о фразе** | `<sentence-id>-note` | `cannot-speak-japanese-note`, `speak-little-japanese-note` |
| **grammar-note паттерна** | `<pattern-id>-note` | `where-question-note`, `ask-language-note` |
| **grammar-note общая** (концепция, не привязана к одному токену/фразе) | смысловой слаг без суффикса | `potential`, `go`, `omit-pronoun` |
| **sentence** | по смыслу, НЕ по ромадзи | `greeting-konnichiwa`, `ask-speak-english`, `cannot-speak-japanese`, `staff-welcome`, `staff-yes-little` |
| **pattern** | что делает | `speak-language` |
| **dialog** | тема | `language-help` |
| **lesson** | тема | `language-barrier` |
| **track** | продающая тема (ответ на «зачем ты здесь») | `communication` |

### Многословные токены
Токен = единица, которую студент учит и произносит целиком, не разбирая.
Застывшие выражения (arigatō gozaimasu, yoroshiku onegaishimasu) — один токен,
а не искусственная нарезка. Нарезаем только то, из чего студент реально
собирает новое (wa, ka, dake, kudasai — конструктор; irasshaimase — монолит).

### Почему у заметок бывает суффикс `-note`
Суффикс -note носят заметки о конкретной сущности (токене, фразе, паттерне) — это делает связь видимой из id и заодно решает коллизии с токенами. Общие концептуальные заметки суффикс не носят

### Омонимы
Разные смыслы — разные id через суффикс смысла: `ka` (вопрос) vs `ka-or` («или»). Одинаковый id обязан значить одну и ту же сущность.

## 3. Типы примеров (GrammarExample)

- **transform** — одна форма реально ПРЕВРАЩАЕТСЯ в другую (добавили ka, сменили время): `hanasu → hanaseru`.
- **contrast** — два равноправных правильных варианта; различие в одном элементе меняет смысл или регистр (минимальная пара): `asa wa kōhī dake / asa dake kōhī`. Различающийся элемент помечаем `isHighlight` в сегментах.
- **dialog-pair** — две реплики ДВУХ людей во времени: `doko desu ka → asoko desu`. Кто есть кто — в translation и note, не в геометрии.
- **wrong-right** — левое НЕ СУЩЕСТВУЕТ в языке, правое — норма: `hanasemasu desu ✗ → hanasemasu ✓`.
- **phrase / plain** — одиночная фраза / просто текст.

Тест при выборе: можно ли произнести обе части одним человеком как «было → стало»? → transform. Это два человека? → dialog-pair. Оба правильны и равноправны? → contrast. Левое — ошибка? → wrong-right.

Правила wrong-right:
- у `wrong` нет `translation` (у несуществующей формы нет перевода); исключение — сознательный «потенциальный перевод»
- `note` всегда объясняет, ПОЧЕМУ неверно — без причины блок учит запрету, с причиной — системе
- right-без-wrong — ошибка (используй phrase); wrong-без-right — допустимо, но редкость
- не тиражировать: заметка из пяти красных блоков учит ошибкам лучше, чем норме

Сегменты contrast: без пробелов по краям romaji/cyrillic — рендер сам вставляет пробел между сегментами. Surface конкатенируется без пробелов.

## 4. Стандарты заметок

- **Автономность.** Заметка токена не знает слова «лекция» и читается из любого места приложения (повторение, избранное). Контекстные разборы конкретной фразы — только в заметках уровня sentence/step.
- **Блоки и темы.** Тему можно сознательно разбивать на несколько блоков подряд
  (например, чтобы не сваливать два примера с длинными details в одну кучу).
  Правила разбиения:
  - первый блок темы несёт summary; блоки-продолжения могут идти без summary
    (details + examples) — отсутствие summary и есть сигнал «это продолжение»;
  - блок-продолжение не начинается с висящего референта («их», «эта четвёрка»,
    «также можно») — либо повтори опору («эту четвёрку направлений…»),
    либо не разбивай;
  - вывод/итог темы («выучив одну — получил вторую») не живёт отдельным
    блоком — он строка details последнего блока темы;
  - тема целиком проходит тест: прочитай её блоки, закрыв остальную заметку, —
    вопросов «о чём это?» быть не должно.
- **Глоссы на месте.** Каждое слово в примере, не входящее в разбираемую фразу, расшифровывается в details этого же блока: `kyō (кё:) — «сегодня»`.
- **Sentence-note разбирает все токены фразы** — по строке на каждый, даже если у токена есть своя большая заметка: здесь описывается его роль в данном контексте.
- **Фраза, упомянутая в тексте, дублируется блоком-примером** (мягкое правило): текст объясняет, блок даёт глазам за что зацепиться.
- **translation vs note.** Перевод — в `translation` формы; `note` — только комментарий/сценарий («типичный ответ прохожего, с жестом»). Не склеивать.
- **Body vs deeper.** Body — все роли слова, которые студент реально встретит (частотное — выше). Deeper — механика «почему так», редкости, прививки. Частотное употребление (напр., отказ через chotto…) — это body, а не deeper.
- **Предохранители.** При написании заметки спросить: какое НЕВЕРНОЕ обобщение сделает читатель? — и добавить строку, которая его ловит («у ka есть другие роли — если она в конце, это вопрос»). Границу темы помечать явно: «разберём отдельно, когда встретится».
- **Один грамматический квант на лекцию.** Остальное — лексика поверх знакомых паттернов.

## 5. Кириллица и переводы

- **Кириллица — строго по Поливанову** (си, ти, дзя, э после твёрдых: дакэ, сукоси, коннитива). Расхождения со слухом («скощи», «даке») не меняют запись — они объясняются в заметке произношения токена.
- **Заметки произношения** — на уровне токена, для слов, где Поливанов заметно расходится со слухом (образцы: sukoshi-note, konnichiwa-note, блок произношения в dake-note).
- **kudasai / onegaishimasu в японском ⇔ «пожалуйста» в переводе.** В обе стороны: есть маркер — есть «пожалуйста»; нет маркера (chotto matte) — нет «пожалуйста», иначе студент сочтёт простую форму достаточно вежливой.
- **desu называем «связкой»**, не «глаголом» и не «глаголом-связкой».
- Русские тексты: «аниме» (не «анимэ»).

## 6. Структура папок

Имя файла описывает **содержимое** (часть речи / грамм-тема), **никогда не урок**. Пул общий: одна сущность может использоваться многими уроками.

```
content/
  tracks.json                 # массив TrackInput
  patterns/patterns.json
  sentences/sentences.json
  dialogs/dialogs.json
  tokens/                     # делим по части речи
    particles.json            # wa, ka, dake
    verbs.json                # hanasemasu, hanasemasen
    nouns.json                # eigo, nihongo…
    expressions.json          # konnichiwa, hai, irasshaimase
    adverbs.json              # sukoshi
  grammar-notes/              # делим по грамм-теме
    particles.json            # wa-note, ka-note, dake-note, go
    verb-forms.json           # potential
    phrase-structure.json     # omit-pronoun, ask-language, *-note фраз
    pronunciation.json        # konnichiwa-note, sukoshi-note
    places.json               # kosoado-place, eki-note, toire-note, konbini-note
    politeness.json           # sumimasen-note, chotto-note, ii-note, arigatou-gozaimasu-note, ieie-note
  lessons/
    language-barrier.json     # один объект LessonInput на файл
```

Загрузчик глобит каждую папку и мержит в один пул (падая на дублях id) — дели файлы как удобно. Начинай с одного файла на коллекцию; дроби по ясной теме (`sentences/greetings.json`, `shopping.json`), когда разрастётся. Файл-по-уроку — запрещено.

## 7. Порядок наполнения (снизу вверх, чтобы ссылки резолвились)

1. **grammar-notes** — переиспользуемые объяснения, ни на что не ссылаются.
2. **tokens** — слова; ссылаются на заметки (`grammarNoteId`).
3. **sentences** — собираются из токенов; тут же заметки уровня фразы.
4. **patterns** — если у семьи фраз есть слот; заметка паттерна.
5. **dialogs** — собираются из фраз.
6. **lessons/** — порядок шагов (teach/assemble/speak/listen/dialog); заметки уровня шага.
7. **tracks.json** — кладём урок в трек, задаём `position`.

Порядок — рекомендация; валидатор всё равно поймает несостыковки.

## 8. Что проверяет утилита

```
npm run content -- check          # целостность ссылок, дубли, глобальная уникальность id
npm run content -- status         # счётчики, сироты, аудио-пробелы
npm run content -- where <id>     # что это за id и кто на него ссылается (ОБЯЗАТЕЛЬНО перед переименованием/удалением)
npm run content -- bundle --lesson <id> --out <dir>   # собрать LessonBundle (preview / вычитка ИИ)
npm run content -- diff           # сравнить content/ с БД
npm run content -- apply          # привести БД к content/ (create/update/delete в транзакции)
```

Прежде чем что-то переименовать или удалить — прогони `where <id>`, чтобы увидеть все ссылки. Правило вечного id важнее удобства.
